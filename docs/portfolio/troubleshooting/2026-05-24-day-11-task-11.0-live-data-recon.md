## 2. Issue 1 — pyiceberg S3 의 `minio` hostname 간헐 DNS 해석 실패

### 증상

로컬 FastAPI `/api/hotspots` 를 **반복 호출하면 첫 요청만 200, 이후 전부 500**:

```
hotspots #1 :  5.86s   HTTP 200   ✅
hotspots #2 :  5.50s   HTTP 500   ❌
hotspots #3 :  1.49s   HTTP 500   ❌
```

`/health` 는 항상 정상 → 프로세스는 살아있음. uvicorn stdout 버퍼링 때문에 처음엔 traceback 이 안 보였고, **`PYTHONUNBUFFERED=1` 재기동 + 포트 점유 좀비 프로세스 정리** 후에야 실제 에러를 포착:

```
OSError: When reading information for key 'warehouse/.../snap-...avro'
in bucket 'seoul-warehouse':
AWS Error NETWORK_CONNECTION during HeadObject operation:
curlCode: 6, Couldn't resolve host name      ← 핵심
```

### 원인

```
.env: MINIO_ENDPOINT=http://minio:9000   (docker-internal endpoint, Day 1 환경 편차)

  docker 내부 ─── compose DNS ────────▶ minio  ✅ 항상 OK
  호스트 FastAPI ── /etc/hosts alias ──▶ minio  ⚠️ 간헐 실패
```

- **DuckDB 쪽** (`configure_duckdb`) 은 `minio_endpoint` 를 S3 SECRET 으로 설정 → 정상 동작.
- **pyiceberg 쪽** (`plan_files()` 의 PyArrow S3FileSystem = AWS C++ SDK) 이 manifest(.avro) 를 읽을 때 `minio` 해석을 **간헐적으로 실패**.
- AWS SDK 의 DNS 해석이 `/etc/hosts` 를 일관되게 참조하지 않거나, FastAPI threadpool 스레드에서 negative 캐시 / race 발생으로 추정.

### 해결 (정찰 시점 임시)

endpoint 를 **`localhost`** (special-cased, 항상 resolve) 로 override:

```bash
MINIO_ENDPOINT=http://localhost:9000 LAKEKEEPER_URL=http://localhost:8181 \
  PYTHONPATH=src uvicorn api.main:app --host 127.0.0.1 --port 8000
```

> **결과** 5회 연속 모두 200 → 500 사라짐. DNS 실패가 endpoint hostname 차이의 증상임이 **확정**.

### ▶ Task 11.1 이월 메모

- 호스트 FastAPI 운영 시 endpoint 를 `localhost`/`127.0.0.1` 로 고정하거나 pyiceberg FileIO 의 S3 설정을 명시 주입.
- Oracle VM 배포 시엔 compose 네트워크 내부라 `minio` 해석이 정상일 가능성 → **환경별 endpoint 분리 설정** 필요.

## 3. Issue 2 — gold small-file accumulation read latency ⭐ (핵심)

### 증상

Issue 1 해결 후 500 은 사라졌으나, 응답 시간이 **5초 ~ 29초로 극심하게 변동**:

```
hotspots #1 :  6.17s    #2 : 28.94s    #3 :  5.05s
hotspots #4 : 26.20s    #5 :  5.02s
              └──────────┬──────────┘
            4초 edge timeout 으론 절대 못 버팀 ❌
```

### 원인 (실측)

**단계별 분해 측정:**

| 단계 | cold | warm |
|---|---:|---:|
| `build_catalog()` | 0.09s | — |
| `load_table()` | 0.43s | — |
| `scan().plan_files()` — manifest 읽기 | **2.29s** | 0.25s |
| DuckDB `read_parquet` — data 읽기 | **2.78s** | — |

**스모킹건:**

```
┌──────────────────────────────────────────────┐
│  data file 개수 :  9,161                       │
│  snapshot 개수  :  3,314                       │
│  총 row 수      :  9,929   →  파일당 약 1 row  │  ⚠️
└──────────────────────────────────────────────┘
```

- streaming 이 5분 micro-batch 로 **tiny parquet 를 9천 개** 쌓은 전형적 **small-file 문제**.
- 병목은 **파일 수에 정비례하는 두 곳**:
  - pyiceberg 의 manifest list 읽기 (2.3s)
  - DuckDB 의 9,161개 파일 S3 read (2.8s)
- 로컬은 ~5s 인데 HTTP 가 **29s 까지 튀는 이유** = 9천 파일을 S3 로 읽을 때 connection 재시도 + (Issue 3 의) 스레드 경합이 겹침.

### 해결 방향 (Task 11.1 이월)

```
rewrite_data_files   :  9,161 작은 파일  →  소수 큰 파일    →  read <0.5s 기대
expire_snapshots     :  3,314 snapshot  →  metadata 축소
```

> ⚠️ **함정** — 기존 `iceberg_maintenance` DAG 의 compaction 대상은 **`silver.hotspot_congestion` 뿐**.
> 느린 **gold(`fact_hotspot_congestion_5min`)는 미포함** (docstring: gold / dim_place 는 "P1B 후 활성화 의무" 로 미뤄둠).
> → **gold compaction task 신규 작성 필요.**

## 4. Issue 3 — FastAPI singleton DuckDB + pyiceberg catalog 의 thread-safety

### 증상

페이지가 두 endpoint 를 **병렬(`Promise.all`)** 로 호출하면 둘 다 **즉시(0.06s) 502**:

```
parallel hotspots :  0.07s   HTTP 502   ❌
parallel chill    :  0.07s   HTTP 502   ❌
```

### 원인

```
api/deps.py
   duck_connection()  ──@lru_cache──▶  단일 DuckDB 연결 객체  ┐
   _catalog()         ──@lru_cache──▶  단일 pyiceberg Catalog ┘
                                              │
        FastAPI sync(def) 엔드포인트          │  여러 스레드가
        = anyio threadpool 에서 동시 실행 ────┘  하나의 객체 공유
                                              │
                                              ▼
                          DuckDB / pyiceberg Catalog 는
                          스레드 동시 접근에 안전하지 않음 → 경합 / 상태 손상
```

> Issue 1(DNS) 과 증상이 겹쳐 보였으나 **별개 뿌리**: Issue 1 = DNS, Issue 3 = thread-safety.

### 해결 방향 (Task 11.1 이월)

- 요청마다 DuckDB `con.cursor()` 사용 (DuckDB 권장 동시성 모델), 또는 연결 / 카탈로그를 **요청 스코프로 분리**.
- 또는 serving 을 live-read 가 아닌 **사전 적재(Postgres/D1) + 단순 select** 로 전환하면 동시성·latency 문제가 **한꺼번에 해소** (§5 참조).

## 5. 아키텍처 인사이트 — Option 5(FastAPI 직결)는 compaction 이 전제

Task 11.0 에서 데이터 source 를 4옵션 trade-off 후 **Option 2(Postgres sync)** 로 정했다가, Day 8 기존 FastAPI(`/api/chill-open`) 발견으로 **Option 5(FastAPI 직결)** 로 pivot 했다. 본 정찰이 그 결정의 **숨은 전제**를 드러냈다.

```
                  compaction 안 함                 compaction 함
                 ┌──────────────┐               ┌──────────────┐
  Option 5       │ live-read     │               │ live-read     │
  (FastAPI 직결) │ 5 ~ 29s  ❌    │   ──그 후──▶   │ < 1s     ✅    │
                 └──────────────┘               └──────────────┘
                        │
                        └─ Edge cache(60s)로도 못 가림 (첫 요청 비용 그대로)
```

- Option 5 의 "live mart read" 는 **compaction 이 선행돼야** 빠르다.
- 즉 **Option 5 vs Option 2 의 진짜 축은 "compaction 했는가"** 였다.
- **순서 결론:**

  ```
  compaction first  →  (필요 시) serving cache
  ```

  compaction 은 어차피 해야 하는 storage 위생 작업이며, **1번 포트폴리오의 "Compaction 도입 예정" 미해결 이슈 closure 서사**(CLAUDE.md §10)와 직결된다.
